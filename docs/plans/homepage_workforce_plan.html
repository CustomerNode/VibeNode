Homepage & Workforce Asset Library

Replace the view mode selector with a proper homepage. Introduce Workforce as the canonical library for managing skills and agent definitions across all of VibeNode.
April 2026Navigation + Workforce + READMEMulti-phase

Overview

VibeNode currently uses a dropdown/modal to switch between four view modes (Grid, List, Workflow, Workforce). This plan replaces that with a homepage — three premium cards that position VibeNode as a platform with three distinct products rather than a session manager with tabs.

Core insight: Claude Code's "skills" and "agents" are the same primitive — a markdown file with instructions. The only difference is invocation: skills inject into the current session, agents spawn a subprocess. VibeNode's Workforce view unifies both into a single asset library where the same .md definition can be invoked either way depending on context.

Why Now — The Convergence Is Happening

The community is already recognizing that agents, skills, and commands are converging toward a single concept. VibeNode's Workforce is the first implementation of what others are only writing about.

Two invocation modes: automatic (Claude decides) and explicit (user calls by name). We complete the convergence by bringing subagents into the same library. The convergence thesis is in the air. Multiple voices are articulating it. But nobody has shipped a unified library with a GUI that treats agents, skills, and pipelines as interchangeable assets from a single collection with switchable invocation modes. That's what Workforce is.

The Format Fragmentation Problem

Today's ecosystem has at least 5 incompatible .md formats that all do the same thing — put instructions in a markdown file:

FormatLocationRequired FieldsUnique Fields
Claude Code Subagents .claude/agents/*.md name, description tools, model, color, maxTurns, memory, isolation, background, permissionMode, skills
Claude Code Skills .claude/skills/*/SKILL.md name, description (recommended) allowed-tools, user-invocable, disable-model-invocation, context, agent, paths, shell, argument-hint
Agent Skills Standard .claude/skills/*/SKILL.md name, description license, compatibility, metadata (cross-platform subset)
gstack Skills .claude/skills/gstack/*/SKILL.md name, description preamble-tier (gstack-only, not recognized by Claude Code)
Plain .md (no frontmatter) .claude/agents/*.md or anywhere Nothing Nothing — entire file is instructions

Workforce's discovery engine reads all of these and normalizes them into a single view. We don't invent a new format — we read what's already there, map the fields we recognize, preserve the rest, and present everything through one UI. The .md file is the universal contract; Workforce is the universal manager.

Design Decisions

  • Grid and List collapse into Sessions. These were never separate "views" — they're display preferences for the same thing. A toggle in the three-dot menu switches between grid and list layout.
  • First load = homepage. Fresh install lands here. After that, the app resumes your last view on startup (stored in localStorage). The homepage is always one click away via two paths:
    • View mode button (sidebar) — takes you to the homepage. The button label reflects which view you're currently in ("Sessions", "Workflow", "Workforce"). When you're on the homepage, it shows "Home" or similar.
    • VibeNode logo (top header — image or text) — hover scales up slightly (transform: scale(1.05) with smooth transition), cursor becomes pointer. Click navigates to the homepage, identical to the view mode button.
    When you pick a view from the homepage, that view is saved as your current location. Next startup restores that view directly — you don't see the homepage again unless you click to it.
  • Live status on each card. Shows real counts (working/idle sessions, tasks in progress, active agents). Makes the homepage useful as a command center, not just a menu screen.
  • Minimal visualization per card. One compact, inline viz that reads at a glance:
    • Sessions — Thin segmented bar, proportional widths by state (working=blue, idle=dim, waiting=orange, sleeping=gray). 4px tall, rounded.
    • Workflow — Slim progress bar showing completion ratio across all tasks. Gradient fill (green to cyan) against a dark track.
    • Workforce — Dot grid, one dot per agent, colored by department. Dense, visual, shows library scale and composition at a glance.
    No axes, no labels, no legends. Pure visual texture that rewards a glance without demanding attention.
  • Premium, modern card design. Large icons, clean typography, subtle gradients. These are the three pillars of VibeNode — they should look like it.

2 — Sessions (Collapsed Grid + List)

Sessions is the traditional session manager — but it's not just a manager. You launch sessions, watch Claude work in real time, answer questions, send commands. It's the full interactive terminal experience.

What Changes

  • View mode workforce (grid) and list merge into a single sessions mode. The sidebar shows session cards or rows depending on a display preference toggle.
  • Display toggle lives in the sidebar three-dot menu: "Display as Grid" / "Display as List". Persisted to localStorage.
  • The setViewMode() function in workforce.js drops 'workforce' and 'list' as separate modes, replaces with 'sessions'. Internal display sub-mode handled separately.
  • Everything else stays the same. Search, sort, new session, live panel, chat — all unchanged.
Migration note: Users with localStorage.viewMode = 'workforce' or 'list' need to be silently migrated to 'sessions' on first load. A one-liner in the init code.

3 — Workforce: The Knowledge Asset Library

Workforce becomes the canonical system for managing Claude Code instruction assets. Instead of the SDK's split concepts, VibeNode treats skills and agents as one thing: a .md file with a role definition that can be invoked either way depending on context.

Bridging the complexity gap: The ecosystem already has assets at two extremes — simple role prompts ("You are a frontend engineer...") and full execution pipelines like gstack (multi-phase workflows with shell scripts, tool permissions, specialist dispatch, telemetry). Workforce handles both natively. The same library, the same UI, the same invocation paths — whether the asset is a one-paragraph persona or a 1000-line review pipeline.

Asset Complexity Tiers

Every asset in the library is a .md file with YAML frontmatter. The format is a superset that spans the full complexity spectrum. Simple assets use minimal frontmatter; complex skills use the full spec. VibeNode reads the frontmatter to determine how to handle each asset.

TierWhat It IsFrontmatterExample
RoleA persona / system prompt. One paragraph to a few pages of instructions.id, name, department, tagsFrontend Engineer, Writing Assistant, any of the 72 built-in agents
SkillA structured workflow with steps, checklists, or output format requirements. May specify tool permissions.+ allowed-tools, versionCode review checklist, PR template generator, documentation auditor
PipelineA full execution pipeline with shell preambles, multi-phase steps, specialist dispatch, external dependencies, and telemetry.+ preamble-tier, dependencies, sourcegstack /review, /qa, /cso, /ship

All three tiers live in the same library, appear in the same UI, and use the same invocation paths. The tier is derived from the frontmatter — not declared explicitly. VibeNode just looks at what's present.

Dual Invocation Model

Every asset — regardless of tier — can be used two ways from the same definition:

ModeHow It WorksWhen It Happens
Agent (default)Catalog injected into every session's system prompt. Claude spawns specialists as subprocesses via Agent tool when it decides to delegate.Automatically, every session. Claude decides when to use them.
Skill (on-demand)User explicitly invokes via Workforce UI click or slash command. The asset's full content is injected into the current session.User triggers it — from Workforce View or via /invoke <id> in chat.

The invocation mode is the same regardless of tier. A simple role prompt and a 1000-line gstack pipeline both get injected the same way — Claude handles the complexity difference because the instructions themselves tell it what to do. The .md is the program.

Discovery — Finding What's Already Installed

On startup (and on-demand via refresh), VibeNode scans for assets across all known locations:

SourcePathWhat's Found
Built-inVibeNode's own default libraryThe 72 built-in role agents (FOLDER_SUPERSET migrated to .md files)
User~/.claude/workforce/User-created or imported assets (global, cross-project)
Project{project}/.claude/workforce/Project-specific assets (override global by ID)
Claude Code Agents~/.claude/agents/, {project}/.claude/agents/Native Claude Code agent definitions
Claude Code Skills~/.claude/skills/Installed skill packs (gstack, custom skills, etc.)

Discovered assets from .claude/agents/ and .claude/skills/ appear in the Workforce library with a badge indicating their source. They're read-only in VibeNode by default — editing creates a workforce-local copy. If you install gstack, its 23+ skills appear in your Workforce automatically.

Invocation — Slash Commands + Visual Picker

Two ways to invoke assets mid-session: text commands and a GUI picker.

  • /invoke <id> or /as <id> — Inject that asset's full content into the current running session
  • /agents — List available assets in a toast or inline display
  • Tab-completion on /invoke showing available asset IDs

Visual hierarchy picker (GUI): A button in the live panel toolbar (or triggered via /team) opens a compact flyout panel showing the full workforce hierarchy as a navigable tree. Click into departments to drill down, click an asset to invoke it. Shows tier badges, source labels, and a preview of the first line of the system prompt. This is the graphical equivalent of /invoke — same injection behavior, but you can browse the tree instead of remembering IDs.

The picker reuses the same hierarchy rendering as Workforce view but in a compact, panel-sized format. Breadcrumb navigation for drilling in/out. Search/filter at the top. Recently used assets pinned at the top of the list.

Invocation works the same for all tiers — simple role or full gstack pipeline:

[Invoking: Pre-Landing Review (gstack-review)]
{full SKILL.md content — steps, checklists, shell blocks, everything}

{user's actual message, if any}

New Session Quick-Pick

When starting a new session (from any view), the input area shows an optional agent selector — a small dropdown or chip bar above the textarea:

  • Recently used — last 3-5 assets you invoked, shown as clickable chips
  • Browse hierarchy — opens the same flyout picker, lets you navigate the full tree
  • No selection = default — session starts with just the catalog injection (all agents available, no specific persona)

Selecting an asset before your first message means it gets injected as a proper system prompt (not a user message). This gives it full authority — stronger than mid-session /invoke.

4 — Externalized File Format

Move from hardcoded JS (FOLDER_SUPERSET) to on-disk .md files. The format is a superset that accommodates everything from one-paragraph role prompts to full gstack pipelines.

Storage Locations

  • Global library: ~/.claude/workforce/ — available across all projects
  • Per-project overrides: .claude/workforce/ in the project root
  • Hierarchy file: workforce-map.md in either location
  • External skill packs: ~/.claude/skills/*/SKILL.md — auto-discovered (gstack, etc.)
  • Per-project assets override global ones with the same id

Unified Asset File Spec

Only id, name, and department are required. Everything else is optional and additive. Every asset belongs to a department — this is how the hierarchy stays organized.

---
# Required
id: eng-frontend                    # Unique identifier
name: Frontend Engineer             # Display name
department: Engineering             # Department — always required, drives hierarchy placement
tags: [react, typescript, css]      # Searchable tags
icon: code                          # Icon key for UI
active: true                        # Whether included in catalog (default: true)

# Skill tier (optional)
version: 1.0.0                      # Semantic version
allowed-tools: [Bash, Read, Edit]   # Tool permissions

# Pipeline tier (optional)
preamble-tier: 4                    # Shell preamble priority
source: gstack                      # Origin skill pack
dependencies: [bun, playwright]     # External runtime dependencies
---

{body: system prompt, structured instructions, or full pipeline}

Mapping Existing Claude Code Agents & Skills into the Hierarchy

Claude Code agents and skills have different file structures. The discovery engine normalizes both into the workforce format and places them in the hierarchy intelligently.

SourceOn-Disk FormatHow It Maps
Claude Code Agents
.claude/agents/*.md or
{project}/.claude/agents/*.md		 Single .md file per agent. May have frontmatter (name, description, model, color, tools) or may be plain markdown with no frontmatter at all.
  • id = filename without extension (e.g., coder.md becomes coder)
  • name = frontmatter name if present, else title-cased filename
  • allowed-tools = frontmatter tools if present
  • Tier = Role (if no tools/version) or Skill (if tools specified)
  • Body = everything after frontmatter (the system prompt)
Claude Code Skills
.claude/skills/{name}/SKILL.md		 Directory per skill with SKILL.md inside. Frontmatter includes name, description, version, allowed-tools, user-invocable. May have supporting files alongside.
  • id = directory name (e.g., review/ becomes review)
  • name = frontmatter name
  • allowed-tools = frontmatter allowed-tools
  • version = frontmatter version
  • Tier = Skill or Pipeline (if preamble-tier present or shell blocks detected)
Skill Packs
.claude/skills/{pack}/
(e.g., gstack)		 Directory with multiple sub-skill directories, each containing its own SKILL.md. May have a top-level SKILL.md for the pack itself, plus shared config (CLAUDE.md, AGENTS.md).
  • source = pack directory name (e.g., gstack)
  • Each sub-skill gets its own asset, prefixed with pack name (gstack-review, gstack-qa)
  • Auto-grouped under a parent node named after the pack
Plugins
.claude/plugins/*/		 Plugin directories with .claude-plugin/plugin.json, optional agents/ and skills/ subdirs.
  • Agents and skills within plugins are discovered the same way
  • source = plugin name from plugin.json
  • Auto-grouped under a parent node named after the plugin

Hierarchy Placement Rules & Department Assignment

Every asset gets a department — this is required. When discovered assets don't have one explicitly, the discovery engine assigns it based on source:

  • Standalone agents (from .claude/agents/) — department = "Imported Agents". Placed in that branch at root level. User can drag them to any department.
  • Standalone skills (from .claude/skills/{name}/SKILL.md where not part of a pack) — department = "Imported Skills". User can reorganize.
  • Skill packs (from .claude/skills/{pack}/* with multiple sub-skills) — department = pack name (e.g., "gstack"). Get their own top-level branch. Sub-skills nested underneath. Pack structure preserved.
  • Plugins (from .claude/plugins/) — department = plugin name. Same treatment as skill packs.
  • Project-scoped agents (from {project}/.claude/agents/) — department = "Project Agents", distinguished with a badge.

All placement is initial — the user can drag anything to any department after import. Moving an asset updates its department in the map file. Re-discovery doesn't reset positions; it only adds newly found assets to their default locations and flags removed ones.

One hierarchy, everywhere: The department tree is the same everywhere it appears — in Workforce view, in the mid-session hierarchy picker, in the new session quick-pick dropdown, and in the agent catalog injection. There's one tree, one map file, one source of truth. If you rearrange departments in Workforce view, the same structure shows up when you browse agents from inside a running session. This is the same hierarchy with the same departments — Engineering, QA, Product, gstack, Imported Agents, etc.

Handling Assets Without Frontmatter

Some agents (especially hand-written ones in .claude/agents/) have no frontmatter at all — just plain markdown instructions. The discovery engine handles this gracefully:

  • id = filename without extension
  • name = title-cased, hyphens/underscores replaced with spaces (e.g., code_reviewer.md becomes "Code Reviewer")
  • department = auto-assigned based on source location (see placement rules above)
  • Tier = Role (assumed simplest tier)
  • Entire file content = the system prompt body
  • No allowed-tools restriction (inherits session defaults)

This means any .md file can become a workforce asset — you don't need to add frontmatter for it to work. Frontmatter just gives you more control over how it appears and behaves in the library.

5 — Workforce UI Features

The Workforce View is the management interface for the asset library.

Activate / Deactivate

Toggle agents active or inactive. Only active agents are written to the catalog injection.

Edit Inline

Click any asset to edit directly. External assets (gstack, etc.) create a local override copy on first edit.

Import

Upload .md files, paste from clipboard, or auto-discover from .claude/agents/, .claude/skills/, and installed skill packs.

Export & Share

Download individual assets or department bundles as .md files. Copy-paste friendly.

Launch Session

Click into any asset and "New Session" to start with that content pre-loaded. Works for all tiers.

Hierarchy Management

Drag-and-drop rearrangement. Create departments, move agents. Updates the map file on disk.

Tier Badges & Source Labels

Each asset shows its tier — Role, Skill, or Pipeline — and source (gstack, built-in, user).

Dependency Warnings

Pipeline-tier assets may require external tools (Bun, Playwright). Shows dependency status and installation guidance.

6 — How the Asset Library Integrates with Each View

The Workforce asset library isn't siloed — it powers all three views. Here's how the same .md assets surface in each context.

Sessions View

This is where you run and interact with Claude Code sessions. The asset library integrates here in two ways:

  • Agent catalog injection (automatic). Every session started from Sessions view gets the full agent catalog injected into its system prompt — same as today. Claude can see all active agents and spawn them as subprocesses when it decides to delegate. This happens invisibly; the user doesn't need to do anything.
  • Slash command invocation (on-demand). While chatting in the live panel, type /invoke <id> to inject any asset mid-conversation. This lets you switch personas or activate a pipeline without leaving the session. Tab-completion shows available IDs. /agents lists what's available. /team opens an inline picker.
  • Session start with preset. The "New Session" flow could offer a quick-pick dropdown of recently used or pinned assets, so you can start a session already scoped to a role without going to Workforce first.

Workflow View

This is the hierarchical task board. The asset library integrates here at the task-to-session boundary:

  • Task-level agent assignment. When creating or editing a task, you can assign a workforce asset to it. When a session is spawned from that task, the asset's system prompt is injected alongside the existing task context (breadcrumb path, sibling tasks, parent description). The agent scoping stacks with the task scoping.
  • Agent catalog still applies. Sessions launched from Workflow still get the full catalog injection, so Claude can delegate across the workforce even when scoped to a specific task. The task-level asset just sets the primary persona.
  • Pipeline assets for task types. A task like "Security audit before release" could be assigned the gstack-cso pipeline asset. When a session launches from that task, it gets the full CSO pipeline workflow injected — so the session runs through the security audit steps automatically, scoped to the task's context.

Workforce View

This is the asset library itself — the management and launch interface:

  • Browse, edit, organize. The full hierarchy of assets, with tier badges, source labels, activation toggles. This is where you manage what's available across the other two views.
  • Launch sessions directly. Click into any asset, hit "New Session", and you're in a session with that asset's full content pre-loaded as the system prompt. For pipeline assets, this means the session starts executing the pipeline immediately.
  • Discovery and import. See everything that's installed — built-in agents, user-created assets, Claude Code agents/skills, gstack packs. Import new assets, rearrange the hierarchy, export bundles to share.
  • Activation controls what the other views see. Only active assets appear in the catalog injection (Sessions) and the task assignment picker (Workflow). Deactivating an asset removes it from those contexts without deleting it.
The pattern: Workforce is where you manage assets. Sessions is where you use them interactively (via catalog injection and slash commands). Workflow is where you assign them to tasks so sessions spawn with the right context. All three views read from the same library — changes in Workforce are immediately reflected everywhere.

9 — Risks & Watchouts

Mid-Session Injection Authority

When you /invoke in a running session, the skill prompt arrives as a user message, not a system prompt. Claude may treat it with less authority. Needs testing.

Catalog Size Scaling

72 built-in + user + discovered packs could get large. Only write active agents. For pipeline-tier, write summary/pointer not full content.

Pipeline Dependencies

gstack skills require Bun, Playwright, etc. VibeNode surfaces warnings but can't install them. Need clear UX.

External Asset Versioning

gstack updates via git. Local overrides always win, show "update available" badge when upstream changes.

Hierarchy File Conflicts

If both global and per-project maps exist, per-project extends global, doesn't replace.

SKILL.md Format Compatibility

gstack uses name not id, has fields we don't use (preamble-tier). Discovery engine normalizes into our superset without losing information.
