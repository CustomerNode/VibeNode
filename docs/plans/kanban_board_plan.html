Executive Summary

This plan adds a fourth view mode to ClaudeGUI — a Kanban-style board that lets users organize work into configurable workflow columns. Unlike the existing List, Workforce, and Workplace views which are purely session-centric, the Kanban view introduces a new first-class concept: Tasks.

Tasks are arbitrarily nestable units of work that live in columns (statuses). Each task can contain subtasks (which can themselves contain subtasks — unlimited depth), and each task or subtask can have linked sessions spawned directly from within it. Sessions stay on disk — the DB just tracks the link. This bridges the gap between "what are we building" (tasks) and "how we're building it" (Claude sessions).

Because task data is critical and persistent, we introduce ClaudeGUI's first relational database layer with a pluggable backend supporting both SQLite (local, zero-config) and Supabase (cloud, collaborative). A clean repository abstraction ensures the rest of the codebase is backend-agnostic.

Key Design Decisions Tasks are scoped to projects via foreign key. Sessions stay on disk where the daemon manages them — the kanban DB only stores a thin link table (task_sessions) mapping session IDs to task IDs. No session data enters the database. Orphaned links are harmless — chats are disposable, the task structure is what matters. Status propagation flows upward automatically. The column set is user-configurable but ships with sensible defaults.

// static/js/time-utils.js — extracted from sessions.js for reuse
// Both sessions.js and kanban.js import this

function shortDate(dateStr) {
    const d = new Date(dateStr);
    if (isNaN(d)) return dateStr;
    const now = new Date();
    const today = new Date(now.getFullYear(), now.getMonth(), now.getDate());
    const target = new Date(d.getFullYear(), d.getMonth(), d.getDate());
    const diffDays = Math.round((today - target) / 86400000);

    let h = d.getHours();
    const ampm = h >= 12 ? 'PM' : 'AM';
    h = h % 12 || 12;
    const min = String(d.getMinutes()).padStart(2, '0');
    const time = h + ':' + min + ' ' + ampm;

    if (diffDays === 0) return time;                // "2:34 PM"
    if (diffDays === 1) return 'Yesterday';          // "Yesterday"

    const dayNames = ['Sunday','Monday','Tuesday','Wednesday',
                      'Thursday','Friday','Saturday'];
    if (diffDays >= 2 && diffDays <= 6) return dayNames[d.getDay()];

    const months = ['Jan','Feb','Mar','Apr','May','Jun',
                    'Jul','Aug','Sep','Oct','Nov','Dec'];
    if (d.getFullYear() !== now.getFullYear())
        return months[d.getMonth()] + ' ' + d.getDate() + " '" + String(d.getFullYear()).slice(-2);
    return months[d.getMonth()] + ' ' + d.getDate(); // "Mar 18"
}

Browser History & Navigation Integration

Every drill-down pushes a state entry to the browser history via history.pushState(). This means the browser's native Back, Forward, and Refresh buttons all work naturally.

URL Scheme

ViewURL HashExample
Board (top level)#kanban/#kanban

Nested subtask#kanban/task/{id}/#kanban/task/d4e5f6

Note: the URL only contains the current task ID — the breadcrumb is reconstructed from the parent_id chain on load. This keeps URLs short and shareable.

// kanban.js — history integration

function navigateToTask(taskId) {
    const state = { view: 'kanban', taskId };
    history.pushState(state, '', `#kanban/task/${taskId}`);
    renderTaskDetail(taskId);
}

function navigateToBoard() {
    const state = { view: 'kanban', taskId: null };
    history.pushState(state, '', '#kanban');
    renderKanbanBoard();
}

// Handle browser back/forward buttons
window.addEventListener('popstate', (e) => {
    if (e.state?.view === 'kanban') {
        if (e.state.taskId) {
            renderTaskDetail(e.state.taskId);
        } else {
            renderKanbanBoard();
        }
    }
});

// Handle page refresh — parse hash on load
function restoreFromHash() {
    const match = location.hash.match(/^#kanban\/task\/(.+)$/);
    if (match) {
        const taskId = match[1];
        setViewMode('kanban');
        renderTaskDetail(taskId);   // rebuilds breadcrumb from parent chain
    } else if (location.hash === '#kanban') {
        setViewMode('kanban');
        renderKanbanBoard();
    }
}

// Breadcrumb click — also pushes to history
function onBreadcrumbClick(taskId) {
    if (taskId === null) {
        navigateToBoard();           // clicked "Board" crumb
    } else {
        navigateToTask(taskId);      // clicked an ancestor crumb
    }
}

Task & Subtask Hierarchy

Tasks use an adjacency list model (each row has a parent_id FK pointing to its parent task, or NULL for root tasks). This is the simplest model that supports unlimited nesting while keeping queries straightforward. 


  Project: "ClaudeGUI"
  │
  ├─ [Task] Auth system redesign          ← root task, status: Working
  │  ├─ [Subtask] OAuth2 provider setup     ← child, status: Complete
  │  │  ├─ [Session] "Implement Google OAuth"
  │  │  └─ [Session] "Add GitHub OAuth"
  │  ├─ [Subtask] Session management        ← child, status: Working
  │  │  ├─ [Subtask] Token rotation          ← grandchild, status: Working
  │  │  │  └─ [Session] "Implement refresh tokens"
  │  │  └─ [Subtask] Redis session store     ← grandchild, status: Not Started
  │  └─ [Subtask] RBAC permissions          ← child, status: Not Started
  │
  └─ [Task] Dark mode fixes               ← root task, status: Not Started
     └─ [Session] "Fix contrast issues"
Why Adjacency List over Nested Sets or Materialized Path? For our use case, trees are shallow (typically 2-4 levels), writes are frequent (status changes, task creation), and we rarely need to query all descendants at once. Adjacency lists are the simplest to implement and maintain. If deep recursive queries become a bottleneck, SQLite supports recursive CTEs and Supabase/PostgreSQL has WITH RECURSIVE natively.

Unlimited Depth — No Artificial Limits

There is no depth limit. The adjacency list model with recursive CTEs handles arbitrary nesting cleanly. If a user wants to nest 8 levels deep for a complex feature breakdown, the data model, API, breadcrumb navigation, and UI all support it without restriction.

Status State Machine

Every task (at any depth) has a status. Status transitions follow specific rules. The core design principle is that the human determines when work is done, not the system. By default, the only automatic transition is Not Started → Working (when a session starts or a subtask is entered). All other forward transitions require explicit user action.

Preference: Auto-Advance to Validating A user preference kanban_auto_advance (stored in the preferences table, default: OFF) controls whether the system automatically moves a task from Working → Validating when all children and sessions complete. When OFF (default), the user manually moves tasks forward — they own the workflow. When ON, the system nudges tasks to Validating as a convenience, but Validating → Complete still always requires manual approval.

Hierarchical Propagation Rules

Upward Propagation (child → parent)

  • If any child moves to Working → parent moves to Working (if it was Not Started). Always automatic — this is just acknowledging that work has begun.
  • If all children are Complete or Validating AND no active sessions → parent moves to Validating. Only if kanban_auto_advance is ON (default OFF). Otherwise the user moves it manually.
  • If a child moves from Complete to Remediating → parent moves to Remediating (if it was Complete). Always automatic — regression must be surfaced immediately.
  • Propagation is recursive — it walks up the tree to the root

Downward Propagation (parent → child)

  • No automatic downward propagation. Parent status changes never force child statuses. This prevents surprises — you don't want marking a parent as Complete to silently mark unfinished children as Complete.
  • Exception: If a user drags a parent to "Complete" and not all children are complete, show a confirmation dialog: "3 subtasks are not yet complete. Mark all as complete?"
Important: Validating → Complete is Always Manual The transition from Validating to Complete is never automatic. This is intentional — Validating is the "human review checkpoint" where users confirm the work is truly done. This prevents tasks from auto-completing without human verification.

Session Scoping & Spawner

Sessions (Claude conversations) live on disk, managed by the daemon under ~/.claude/. They are NOT stored in the kanban database. The kanban DB only maintains a lightweight link table (task_sessions) that maps a task ID to a session ID string — just the association, not the session data itself. When you open a task card, the UI reads the linked session IDs from the DB, then fetches their live state from the daemon.

No session data in the database

The task_sessions table is a thin junction table: (task_id, session_id, created_at). Session content, state, and history remain on disk where the daemon manages them. This avoids duplicating data and means the kanban system has zero impact on session storage. If a link becomes orphaned (e.g. the session file is manually deleted), the UI simply shows "Session not found" — no cascade problems, no data corruption. Chats are ephemeral work artifacts; the task structure is what matters.

Session Spawner UX

Each task card (at any nesting level) has a "+ New Session" button. Clicking it:

  1. Opens the standard new-session dialog (model selector, initial prompt input)
  2. Pre-fills the task context: task title and breadcrumb path are included as context
  3. On session creation, writes a task_sessions row linking the session ID to the task ID — this is just a string mapping, not a true FK to a sessions table
  4. The parent task transitions to Working if it was Not Started or Remediating
  5. The session appears both in the task's session list AND in the existing List/Workforce/Workplace views (it's the same on-disk session)

Session Lifecycle Integration

When a session linked to a task changes state (detected via the daemon's existing WebSocket events):

  • Session starts → Task moves to Working (auto-transition)
  • Session goes idle/completes → If all linked sessions are also idle/complete AND all subtasks are Validating/Complete → Task may advance (if auto-advance is on)
  • Session enters Question state → No task status change (it's still in-progress)

Session state is read from the daemon's existing SessionInfo — no duplication of session state in the kanban DB. The kanban system is a consumer of session events, not a manager of sessions.

Unscoped Sessions

Sessions created from the existing UI (not through a task's spawner) remain unscoped — they have no entry in task_sessions. These continue to work exactly as they do today in List/Workforce/Workplace views. Users can optionally attach an existing session to a task later via a context menu action.

Orphaned Links Are Fine

If a session's on-disk files get deleted (manually or via claude session rm), the task_sessions link just becomes a dangling reference. The UI handles this gracefully — it shows "Session not found" or simply omits the entry. No cascade deletion needed. The user can clean up stale links from the task detail view if they want, or just ignore them. Chats are disposable work artifacts — the task tree is what has permanent value.

# app/kanban/ai_planner.py

import claude_code_sdk

class TaskPlanner:
    SYSTEM_PROMPT = """You are a task planning assistant. The user will describe
what they want to accomplish. Break it down into a hierarchical task tree.

Respond with a JSON structure:
{
  "tasks": [
    {
      "title": "...",
      "description": "...",  // brief HTML description
      "subtasks": [           // recursive, arbitrary depth
        { "title": "...", "subtasks": [] }
      ]
    }
  ]
}

Guidelines:
- Each task should be a concrete, actionable unit of work
- Aim for tasks completable in 1-3 sessions
- Use 2-4 levels of nesting max unless the user asks for more
- Include descriptions only when the title isn't self-explanatory
"""

    async def plan(self, user_message: str, history: list) -> dict:
        """Send planning request to Claude, return proposed task tree."""
        messages = history + [{"role": "user", "content": user_message}]

        response = await claude_code_sdk.complete(
            system=self.SYSTEM_PROMPT,
            messages=messages,
            max_tokens=4096
        )

        # Parse JSON from response, validate structure
        return self._parse_task_tree(response)

    async def commit_plan(self, project_id: str, task_tree: dict):
        """Batch-insert the accepted task tree into the database."""
        for task_data in task_tree["tasks"]:
            await self._insert_recursive(
                project_id, parent_id=None, data=task_data
            )

🔍 Session Context Injection

When a session is spawned from within a task, Claude should have awareness of what it's working on — but we must be surgical about what we inject to avoid overflowing the context window.

What Gets Injected (Lightweight Context)

  • Breadcrumb path: The full ancestor chain titles — e.g. "Auth system redesign → Session management → Token rotation"
  • Current task title + description: The task this session is scoped to
  • Sibling tasks: Titles only (not descriptions) of tasks at the same level, with status badges — so Claude knows what else is being worked on in parallel
  • Parent task description: One level up only — gives broader context without going overboard
  • Active work awareness: A snapshot of currently running sessions on related tasks — session status (working/idle) and which task each one is on. This prevents two sessions from unknowingly working on overlapping concerns.

What Does NOT Get Injected

  • ✗ Full task tree of the entire project
  • ✗ Other sessions' conversation history
  • ✗ Descriptions of tasks more than 1 level up
  • ✗ Completed sibling task details (just their titles + "complete" badge)
  • ✗ Tags, status history, or reporting data
  • ✗ Sessions on unrelated tasks (different parent branch)

Context Template

# Injected as a system prompt prefix when starting a task-scoped session

TASK_CONTEXT_TEMPLATE = """You are working on a specific task within a larger project.

## Current Task
**{task_title}**
{task_description}

## Position in Project
{breadcrumb_path}

## Parent Context
{parent_title}: {parent_description}

## Sibling Tasks (same level)
{sibling_list}

## Active Sessions Nearby
{active_sessions}

Focus your work on the current task. The sibling tasks and active sessions
are shown for awareness — coordinate if your work overlaps with theirs.
"""

def build_task_context(task_id: str) -> str:
    """Build a lightweight context string. Target: <600 tokens."""
    task = repo.get_task(task_id)
    ancestors = repo.get_ancestors(task_id)
    siblings = repo.get_children(task.parent_id) if task.parent_id else []
    parent = repo.get_task(task.parent_id) if task.parent_id else None

    breadcrumb = " → ".join([a.title for a in reversed(ancestors)] + [task.title])

    sibling_lines = []
    for s in siblings:
        if s.id == task_id:
            continue
        sibling_lines.append(f"- [{s.status.value}] {s.title}")

    # --- Active work awareness ---
    # Query linked sessions on sibling tasks and the current task's parent,
    # then check the daemon for their live status (working/idle/sleeping).
    active_lines = []
    nearby_task_ids = [s.id for s in siblings if s.id != task_id]
    if task.parent_id:
        nearby_task_ids.append(task.parent_id)
    for tid in nearby_task_ids:
        linked = repo.get_task_sessions(tid)
        nearby_task = repo.get_task(tid)
        for link in linked:
            status = daemon.get_session_status(link.session_id)  # working/idle/sleeping
            if status in ("working", "idle"):
                active_lines.append(
                    f"- [{status}] on \"{nearby_task.title}\" (session {link.session_id[:8]}…)"
                )

    return TASK_CONTEXT_TEMPLATE.format(
        task_title=task.title,
        task_description=task.description or "(no description)",
        breadcrumb_path=breadcrumb,
        parent_title=parent.title if parent else "(root level)",
        parent_description=parent.description if parent else "",
        sibling_list="\n".join(sibling_lines) or "(no siblings)",
        active_sessions="\n".join(active_lines) or "(no active sessions nearby)"
    )
Context Budget: ~600 tokens max The injected context should stay under ~600 tokens. Task descriptions are truncated at 200 characters if longer. Sibling lists are capped at 10 items. Active sessions list is capped at 8 entries (only working/idle shown, sleeping sessions omitted). This leaves the vast majority of Claude's context window for the actual work conversation. The context is injected once at session start via the system prompt — it's not re-injected on every message.

Database Abstraction Layer

We introduce a Repository Pattern with a backend-agnostic interface. All Kanban data access goes through the repository — the rest of the codebase never touches SQL or Supabase APIs directly. 

# app/db/repository.py

from abc import ABC, abstractmethod
from typing import Optional
from dataclasses import dataclass
from enum import Enum

class TaskStatus(Enum):
    NOT_STARTED = "not_started"
    WORKING     = "working"
    VALIDATING  = "validating"
    REMEDIATING  = "remediating"
    COMPLETE    = "complete"

@dataclass

SQLite Backend

The default backend. Zero configuration — the database file is created automatically alongside the existing ClaudeGUI data.

Database Location

# Stored next to existing ClaudeGUI data
~/.claude/gui_kanban.db       # Single SQLite file for all projects

One database file, multiple projects differentiated by project_id foreign keys. This keeps backup simple (one file to copy) and avoids file-per-project sprawl.

Key Implementation Details

  • Uses aiosqlite for async access (pip dependency)
  • WAL mode enabled for concurrent reads during writes
  • Foreign keys enforced via PRAGMA foreign_keys = ON
  • Auto-migration system: version table tracks schema version, migrations run on startup
  • Connection pool: single connection with WAL mode (sufficient for single-user desktop app)
# app/db/sqlite_backend.py — key parts

import aiosqlite
from pathlib import Path

class SQLiteKanbanRepository(KanbanRepository):
    def __init__(self, db_path: Path):
        self._db_path = db_path
        self._conn: Optional[aiosqlite.Connection] = None

    async def initialize(self):
        self._conn = await aiosqlite.connect(self._db_path)
        await self._conn.execute("PRAGMA journal_mode=WAL")
        await self._conn.execute("PRAGMA foreign_keys=ON")
        await self._run_migrations()

    async def get_ancestors(self, task_id: str) -> list[Task]:
        """Walk up the tree using recursive CTE."""
        sql = """
        WITH RECURSIVE ancestors AS (
            SELECT * FROM tasks WHERE id = ?
            UNION ALL
            SELECT t.* FROM tasks t
            JOIN ancestors a ON t.id = a.parent_id
        )
        SELECT * FROM ancestors WHERE id != ?
        """
        async with self._conn.execute(sql, (task_id, task_id)) as cur:
            return [self._row_to_task(row) async for row in cur]

Database Migration (SQLite ↔ Supabase)

Switching backends must be lossless. All tasks, tags, status history, session link mappings, and column config must transfer cleanly in both directions. (Session link mappings are just ID strings — the actual session data stays on disk and is unaffected by database migration.) The user should never have to choose a backend permanently.

Migration Engine

# app/db/migrator.py

class BackendMigrator:
    """Lossless transfer between any two KanbanRepository backends."""

    async def export_all(self, source: KanbanRepository) -> dict:
        """Dump every table into a portable dict. Order matters for FK integrity."""
        return {
            "preferences":    await source.get_all_preferences(),
            "board_columns":  await source.get_all_columns(),
            "tags":           await source.get_all_tags(),
            "tasks":          await source.get_all_tasks_ordered(),  # depth-first: parents before children
            "task_sessions":  await source.get_all_task_sessions(),
            "task_tags":       await source.get_all_task_tags(),
            "status_history": await source.get_all_status_history(),
        }

    async def import_all(self, target: KanbanRepository, data: dict):
        """Write into target. Topological order ensures FK constraints pass."""
        await target.upsert_preferences(data["preferences"])
        await target.upsert_columns_bulk(data["board_columns"])
        await target.upsert_tags_bulk(data["tags"])

        # Tasks in depth-first order (parents first) so FK parent_id resolves
        for task in data["tasks"]:
            await target.upsert_task(task)

        # Link tables after all tasks exist
        await target.upsert_task_sessions_bulk(data["task_sessions"])
        await target.upsert_task_tags_bulk(data["task_tags"])
        await target.upsert_status_history_bulk(data["status_history"])

    async def switch_backend(self, current: KanbanRepository,
                              target: KanbanRepository) -> bool:
        """Full backend switch with rollback safety."""
        # Step 1: Export from current
        data = await self.export_all(current)
        record_count = sum(len(v) for v in data.values())

        # Step 2: Initialize target (creates schema if needed)
        await target.initialize()

        # Step 3: Import into target
        await self.import_all(target, data)

        # Step 4: Verify counts match
        verify = await self.export_all(target)
        verify_count = sum(len(v) for v in verify.values())

        if verify_count != record_count:
            raise MigrationError(
                f"Verification failed: {record_count} records exported, "
                f"{verify_count} found in target. Old backend untouched."
            )

        # Step 5: Only now update the active backend preference
        # The old backend data is NEVER deleted — it stays as a backup
        return True
Rollback Safety: Old Backend Is Never Destroyed When switching from SQLite → Supabase, the gui_kanban.db file stays on disk untouched. When switching from Supabase → SQLite, the Supabase tables stay in place. If anything goes wrong, the user just flips the preference back and they're on their old backend with zero data loss. We verify row counts on the target before declaring success.

For the owner field, we attempt to auto-detect identity in this order:

  2. git config user.name — fallback if no email
  3. OS username (os.getlogin() / %USERNAME%) — last resort
  4. Manual entry in settings — always overrides auto-detection

The detected value is pre-filled in the settings field. Users can override it. The value is stored in the preferences table as kanban_user_identity.

Drag & Drop Implementation

Uses the native HTML5 Drag and Drop API (consistent with the existing Workplace view's drag approach). Task cards have draggable="true". Columns act as drop zones.

  • Between columns: Changes status. Triggers state machine validation (e.g., can't drag from Not Started directly to Complete — must go through Working first, unless explicitly overridden with Shift+drag).
  • Within columns (manual-sort only): Updates the position integer. One UPDATE per drag — task gets a position value between its new neighbors. Only enabled when column's sort_mode is "manual". Date-sorted columns don't support drag reorder.

async def on_session_state_change(session_id: str, new_state: SessionState):
    # Check if this session is linked to a task
    task_id = await repo.get_session_task(session_id)
    if not task_id:
        return

    task = await repo.get_task(task_id)

    if new_state in (SessionState.STARTING, SessionState.WORKING):
        # Session became active → task should be Working
        if task.status in (TaskStatus.NOT_STARTED, TaskStatus.REMEDIATING):
            await transition_task(task_id, TaskStatus.WORKING)

    elif new_state == SessionState.IDLE:
        # Session went idle → check if ALL sessions are idle
        await maybe_advance_to_validating(task_id)

Status Propagation Engine

# app/kanban/state_machine.py

async def transition_task(task_id: str, new_status: TaskStatus):
    """Transition a task and propagate to ancestors."""
    task = await repo.get_task(task_id)
    old_status = task.status

    # Validate transition
    if not is_valid_transition(old_status, new_status):
        raise ValueError(f"Cannot go from {old_status} to {new_status}")

    # Update this task
    await repo.update_task(task_id, status=new_status.value)

    # Emit WebSocket event
    socketio.emit('kanban_task_updated', {'task': task_to_dict(task)})

    # Propagate upward
    if task.parent_id:
        await propagate_status_upward(task.parent_id)


async def propagate_status_upward(task_id: str):
    """Re-evaluate parent status based on all children."""
    task = await repo.get_task(task_id)
    children = await repo.get_children(task_id)
    sessions = await repo.get_task_sessions(task_id)

    all_children_done = all(
        c.status in (TaskStatus.COMPLETE, TaskStatus.VALIDATING)
        for c in children
    )
    any_child_working = any(
        c.status == TaskStatus.WORKING for c in children
    )
    has_active_sessions = check_active_sessions(sessions)

    # Determine appropriate parent status
    if any_child_working or has_active_sessions:
        if task.status == TaskStatus.NOT_STARTED:
            await transition_task(task_id, TaskStatus.WORKING)
    elif all_children_done and not has_active_sessions:
        if task.status == TaskStatus.WORKING:
            await transition_task(task_id, TaskStatus.VALIDATING)

Decided Questions & Design Choices

✓ Resolved Decisions

  • Task descriptions → Rich Text (Quill RTE). Stored as HTML in the description column. Quill provides a lightweight, embeddable editor with formatting toolbar. No Markdown parsing needed — what you see is what's stored.
  • Cross-project tasks → No. Tasks are always strictly scoped to one project via project_id FK. No cross-project spanning.
  • Session auto-attachment → Yes. When a session's conversation mentions a task title, suggest auto-linking via a non-intrusive toast notification: "Link this session to 'Token rotation'?" Click to link, dismiss to ignore.
  • Tasks are NEVER deleted. Completed tasks persist forever. Pagination handles scale — the Complete column loads one page at a time. Historical data is sacred.
  • Time tracking → Yes. Every status transition is recorded in task_status_history with timestamps. Reports surface time-in-status, cycle time, and lead time. History viewable in task drill-down.
  • Labels/tags → Yes. Arbitrary text tags with autocomplete. As you type a tag, existing tags (scoped to project) are suggested via a typeahead dropdown. Tags table tracks usage_count for relevance-ranked suggestions. Tags filterable on the board and broken down in reports.
  • Task ownership → Claimable. Each task has an owner field. Binary claim/unclaim. Identity auto-detected from git config email, overridable in settings. Tracked in status history for reporting.
  • Ordering → Hybrid per-column sorting. Each column has a configurable sort_mode. Manual-sort columns (Not Started, Working) use gap-numbered position integers for drag-based priority ordering — one UPDATE per reorder, no linked list complexity. Automatic-sort columns (Validating, Remediating, Complete) sort by date_entered — most recently transitioned items first. Users can toggle sort mode per column via a dropdown.
  • Tag filtering → Board-level. A filter bar in the board header lets users multi-select tags. When active, all columns filter to show only tasks matching ALL selected tags. Filter state saved in sessionStorage.
  • New project initialization → Automatic. First access to any project's Kanban board auto-creates the 5 default columns with appropriate sort modes. No setup wizard, no error screen. Empty board shows a friendly "Create your first task" prompt.

Handled Edge Cases

  • Orphaned session links: If a session's on-disk files get deleted, the task_sessions row becomes a dangling reference. UI shows "not found" or omits it — no cascade, no data corruption. User can clean up stale links manually or ignore them.
  • Circular parent references: Prevented at the API layer — set_parent() validates the new parent is not a descendant.
  • Concurrent status updates: SQLite WAL mode for concurrent reads, asyncio serialized writes. Supabase uses row-level locking.
  • Project switch: Board re-renders with new project's tasks. Pagination resets.
  • Empty board / new project: First access auto-creates default columns. Friendly empty state with "Create your first task" prompt. No setup wizard needed.
  • Position gap exhaustion (manual-sort columns): If two adjacent tasks have consecutive position integers (no room to insert between), the column's positions are re-numbered in a single batch UPDATE (1000, 2000, 3000...). This is rare and invisible to the user.
  • Huge Complete column: Paginated — default 50 tasks per page, "Load more" button. Sorted by date_entered DESC (most recently completed first) — stable, indexed, no DLL walk needed.
  • Sort mode toggle: Switching a column from manual to date_entered is non-destructive — position integers are preserved. Switching back to manual restores the previous drag order instantly.
  • Tag filter with empty results: If no tasks match the active tag filter in a column, the column shows "No tasks match filters" instead of disappearing entirely.

Per-Column Sorting Strategies

Not all columns should order tasks the same way. Manual-sort columns (Not Started, Working) use a position integer so the user can drag-reorder to express relative priority. Automatic-sort columns (Validating, Remediating, Complete) sort by when the task entered that status — no drag reorder needed.

Available Sort Modes (user-toggleable per column)

  • manual — ORDER BY position ASC. Drag-reorder enabled. Position = implicit priority. Gap-numbered integers (1000, 2000, 3000...) for cheap insertion.
  • date_entered — Sort by task_status_history.changed_at (when the task entered its current status). Drag reorder disabled.
  • date_created — Sort by tasks.created_at. Drag reorder disabled.
  • alphabetical — Sort by tasks.title. Drag reorder disabled.

                           parent_id: str = None, page: int = 1,

        "alphabetical": "t.title",

Manual Reorder — Gap-Numbered Integers

# Drag task B between tasks at position 2000 and 3000:
# → SET position = 2500. One UPDATE. Done.
# If gap exhausted (two adjacent integers), re-number the column:
# → positions become 1000, 2000, 3000... (rare, batch UPDATE)

async def reorder_task(repo, task_id: str, after_id: str, before_id: str):
    """Move task between two neighbors. Single UPDATE in 99% of cases."""
    after_pos  = await repo.get_position(after_id)  if after_id  else 0
    before_pos = await repo.get_position(before_id) if before_id else after_pos + 2000

    new_pos = (after_pos + before_pos) // 2

    if new_pos == after_pos or new_pos == before_pos:
        # Gap exhausted — re-number entire column (rare)
        await _renumber_column(repo, task.project_id, task.status)
        return await reorder_task(repo, task_id, after_id, before_id)

    await repo.update_task(task_id, position=new_pos)  # single UPDATE

URL Auto-Correction

// Frontend: normalize URLs before rendering as clickable links

function resolveVerificationUrl(url) {
    if (!url) return null;
    // Already absolute — use as-is
    if (url.startsWith('http://') || url.startsWith('https://')) return url;
    // Relative path — prepend dev server base
    const base = 'http://localhost:5050';
    return base + (url.startsWith('/') ? url : '/' + url);
}
